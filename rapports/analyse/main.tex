\section{Introduction}

Le présent rapport détaille la composante \texttt{Analyse} relative au projet \textbf{SAÉ S3.02 - Labyrinthe}. Ce projet s'inscrit dans le cadre de la formation BUT Informatique et vise à développer les compétences en modélisation, conception orientée objet et travail collaboratif.

L'application développée propose un jeu de labyrinthe avec deux modes distincts : un \textbf{mode libre} permettant de jouer sur des labyrinthes générés aléatoirement, et un \textbf{mode progression} offrant une expérience structurée avec sauvegarde de l'avancement du joueur à travers différentes étapes de difficulté croissante.

Notre approche de modélisation repose sur les standards UML et suit une méthodologie itérative. Nous avons privilégié une démarche collaborative impliquant l'ensemble des membres de l'équipe, ainsi que des validations régulières auprès de nos enseignants spécialisés en analyse et en qualité de développement.

Le présent rapport aborde l'organisation interne de notre équipe ainsi que notre approche de modélisation des exigences du client, structurée comme suit :

\begin{enumerate}
    \item Diagrammes de cas d'utilisation
    \item Fiches descriptives
    \item Prototype d'interface
    \item Diagramme de classe
\end{enumerate}

Il est entendu que chaque section est itérative et a fait l'objet de plusieurs cycles de révision pour garantir la cohérence et la complétude de la modélisation.

\subsection{Analyse du diagramme de cas d'utilisation}

Le diagramme présenté ci-dessus modélise l'ensemble des interactions fonctionnelles du système Labyrinthe. Il identifie \textbf{sept cas d'utilisation principaux} répartis entre deux acteurs : l'Utilisateur et le Système.

\subsubsection{Acteurs identifiés}

\begin{itemize}
    \item \textbf{Utilisateur} : acteur principal, représente le joueur interagissant avec l'application (lancement du jeu, déplacement dans le labyrinthe, consultation de la progression)
    \item \textbf{Système} : acteur secondaire représentant les processus automatiques de persistance des données (sauvegarde de la progression sans intervention explicite de l'utilisateur)
\end{itemize}

La présence d'un acteur "Système" se justifie dans notre contexte par la nécessité de modéliser explicitement les opérations de persistance automatique. Cette modélisation permet de distinguer clairement les actions intentionnelles de l'utilisateur des mécanismes de sauvegarde implicites déclenchés par le système après chaque action significative (complétion d'un défi, modification du profil).

\subsubsection{Description des cas d'utilisation}

Le diagramme couvre intégralement le \textbf{mode progression}, qui constitue le cœur fonctionnel de l'application :

\begin{enumerate}
    \item \textbf{Jouer} : point d'entrée du jeu (appuyer sur jouer)
    \item \textbf{Lancer en mode progression} : point d'entrée du mode progression, incluant la saisie du nom du joueur
    \item \textbf{Charger profil} : reprise d'une progression existante pour un joueur connu
    \item \textbf{Commencer une partie} : démarrage d'un labyrinthe choisi : en mode libre, on choisit la taille et le pourcentage de murs et en mode progression, on choisit un défi qui définira le labyrinthe selon ses attributs.
    \item \textbf{Afficher la progression} : consultation du tableau de bord avec étapes et défis complétés, score, caractéristiques
    \item \textbf{Se déplacer dans le labyrinthe} : déplacement case par case du joueur (cas d'utilisation détaillé dans la fiche "Déplacer joueur")
    \item \textbf{Sauvegarder la progression} : persistance automatique de l'état du joueur (acteur Système)
\end{enumerate}

\subsubsection{Clarifications}

Afin de respecter une stricte conformité aux besoins client du sujet, le diagramme de cas d'utilisation reflète les choix de modélisation suivants :

\begin{itemize}
    \item \textbf{Distinction entre "Charger profil" et "Lancer en mode progression"} : Le premier cas concerne exclusivement la reprise d'un profil existant avec sa progression sauvegardée. Le second englobe l'initialisation du mode progression (saisie du nom, vérification d'existence, création ou chargement du profil). "Charger un profil" dépend donc de "Lancer en mode progression".

    \item \textbf{Attribution de la sauvegarde au "Système"} : Conformément aux standards UML, les opérations automatiques de persistance sont modélisées par un acteur secondaire. Cette distinction clarifie que la sauvegarde intervient sans action explicite de l'utilisateur après chaque complétion de défi ou modification significative de l'état du joueur.

    \item \textbf{"Jouer" et "Se déplacer dans le labyrinthe"} : Le cas d'utilisation "Jouer" représente l'action de démarrer le jeu (fonctionnalité "Jouer"). "Se déplacer dans le labyrinthe" détaille l'action atomique de déplacement case par case, qui est l'interaction principale durant une partie en cours (labyrinthe). Jouer est inclus car c'est un cas d'utilisation ayant été identifié en TD.

    \item \textbf{Mode libre} : Bien que non représenté dans ce diagramme centré sur le mode progression, le mode libre constitue un parcours alternatif du système. Il suit un flux simplifié sans authentification ni persistance : l'utilisateur sélectionne le mode libre, paramètre les caractéristiques du labyrinthe (largeur, hauteur, pourcentage de murs), puis lance directement une partie sur un labyrinthe généré aléatoirement.  Tous les cas d'utilisation supportent déjà ce mode, ayant été pensés pour cela ("Commencer une partie"). L'absence de sauvegarde distingue fondamentalement ce mode du mode progression. Ce mode sera plus développé dans la partie \textbf{Prototypes}.
\end{itemize}

Le prototype couvre \textbf{huit écrans distincts} répartis entre deux parcours utilisateur : le mode progression et le mode libre. Ces parcours partagent un point d'entrée commun (le menu de choix du mode) avant de diverger selon le mode choisi par l'utilisateur.

\textbf{Parcours utilisateur -- Mode Progression (6 écrans) :}

\begin{enumerate}
    \item \textbf{Menu d'accueil} : point d'entrée de l'application avec les options "Jouer" et "Quitter"
    \item \textbf{Choix du mode} : sélection entre "Mode Libre" et "Mode Progression"
    \item \textbf{Saisie du nom} : identification ou création du profil joueur
    \item \textbf{Gestion du profil existant} : choix entre "Nouvelle partie" et "Reprendre" si un profil existe
    \item \textbf{Sélection étape/défi} : tableau de progression organisé en trois étapes avec défis numérotés, utilisant un code couleur pour indiquer l'état de complétion / accès. L'utilisateur doit cliquer pour sélectionner son défi.
    \item \textbf{Interface de jeu} : labyrinthe avec déplacement du joueur (carré vert) vers la sortie (carré rouge)
\end{enumerate}

\textbf{Parcours utilisateur -- Mode Libre (4 écrans) :}

\begin{enumerate}
    \item \textbf{Menu d'accueil} : point d'entrée de l'application
    \item \textbf{Choix du mode} : sélection entre "Mode Libre" et "Mode Progression"
    \item \textbf{Configuration du labyrinthe} : paramétrage des dimensions (taille, largeur) et du pourcentage de murs
    \item \textbf{Interface de jeu} : labyrinthe généré aléatoirement selon les paramètres choisis
\end{enumerate}

Ces parcours respectent les exigences client avec une progression logique et sans étapes superflues.

\subsection{Structure générale}

L'architecture se décompose en trois niveaux conceptuels :

\begin{enumerate}

\item \textbf{Niveau contrôleur} : \texttt{Main} orchestre le flux général (lancement du jeu, sélection du mode, interaction avec la base de données)
\item \textbf{Niveau logique métier} : \texttt{GameMode}, \texttt{Maze}, \texttt{Player}, \texttt{PlayerProgress}, \texttt{Level}, \texttt{Challenge} encapsulent les règles métier et l'état de la partie
\item \textbf{Niveau persistance} : \texttt{PlayerDatabase} isole les opérations de sauvegarde et de chargement
\end{enumerate}

Cette séparation en couches garantit une maintenance facilitée et une testabilité accrue : chaque niveau peut être testé indépendamment, réduisant ainsi les risques d'anomalies et facilitant l'évolution future du système.

\subsection{Conformité avec la méthode vue en cours}

La méthodologie vue en \texttt{Analyse} préconise l'ordre suivant :

\begin{enumerate}
\item Une classe \texttt{Main} est créée
\item Elle comporte une fonction par cas d'utilisation
\item Pour chaque cas d'utilisation, on enrichit le diagramme de façon à la supporter
\end{enumerate}

Le \texttt{Main} contient bien une fonction par cas d'utilisation et le diagramme supporte chacune ; la méthode est respectée.

\subsection{Clarifications}

Afin d'être le plus concis possible, seules les méthodes pouvant porter à confusion seront décrites. Les méthodes les plus logiques réalisent le traitement demandé par le sujet et n'ont donc pas besoin d'une explication à notre niveau d'abstraction.

\tsecnonum{Les étapes et défis}

\info{Il est entendu qu'un \texttt{Level} correspond à une \textit{Étape}. La traduction littérale de l'anglais aurait donné \texttt{Stage} ce qui rentrerait en conflit avec l'objet \texttt{Stage} de JavaFX.}
\info{Un \texttt{Challenge} est un défi (traduction littérale de l'anglais).}

Il y a précisément 3 étapes et 3 défis (information confirmée en ayant demandé aux professeurs), les structures pour les contenir ont donc cette taille fixe en conséquence.

Un objet \texttt{Challenge} comporte tous les attributs demandés dans le sujet afin de satisfaire au mieux la demande.

La méthode \texttt{isCompleted(): boolean} de \texttt{Level} retourne, conformément au sujet, \texttt{True} si au moins un de ses défis est complété. Elle servira donc à déterminer le \texttt{getHighestLevel()} de \texttt{Player}.

Enfin, les méthodes pour accéder aux étapes possibles | validées ne sont pas mentionnées dans le diagramme de classes car elles seraient très semblables à des \textit{getters} n'étant pas demandés dans le plan.

Il en va de même pour les méthodes de modification (retenter un défi) qui seraient semblables à des \textit{setters}.

\tsecnonum{Le type \texttt{GameMode}}

La fonction \texttt{startGame()} retourne un type \texttt{GameMode}, qui peut aussi bien être le mode libre que le mode progression (type abstrait hérité). Elle est conçue pour être appelée depuis le menu de choix du mode (voir prototypes). 

Ce type abstrait permet un traitement unifié des deux modes de jeu pour les opérations communes (par exemple, \texttt{movePlayerPosition}), évitant ainsi les duplications de code.

Elle sert donc à démarrer une session dans un mode, quel que soit le mode choisi, et couvre naturellement le cas d'utilisation "Lancer mode progression".

\tsecnonum{L'initialisation de \texttt{Player} dans \texttt{ProgressionMode}}
Dans le cas d'un lancement en mode progression, le joueur est dirigé vers une interface de saisie du nom. Une fois le nom validé, la fonction

\texttt{loadPlayer(name: String): Player} est invoquée dans le constructeur \texttt{ProgressionMode} afin de :
\begin{itemize}
\item Charger le profil existant si un joueur avec ce nom est enregistré dans la base de données, permettant ainsi de reprendre la progression sauvegardée
\item Créer un nouveau profil si aucun joueur avec ce nom n'existe, assurant une expérience fluide pour les nouveaux joueurs
\end{itemize}

\texttt{ProgressionMode} prendra donc en attribut le \texttt{Player} retourné.

Cette conception garantit une gestion cohérente et efficace du cycle de vie des profils joueurs, les utilisant uniquement dans le mode progression.
