\section{Introduction}

Le présent rapport détaille la composante \texttt{Analyse} relative au projet \textbf{SAÉ S3.02 - Labyrinthe}. Ce projet s'inscrit dans le cadre de la formation BUT Informatique et vise à développer les compétences en modélisation, conception orientée objet et travail collaboratif.

L'application développée propose un jeu de labyrinthe avec deux modes distincts : un \textbf{mode libre} permettant de jouer sur des labyrinthes générés aléatoirement, et un \textbf{mode progression} offrant une expérience structurée avec sauvegarde de l'avancement du joueur à travers différents niveaux de difficulté croissante.

Notre approche de modélisation repose sur les standards UML et suit une méthodologie itérative. Nous avons privilégié une démarche collaborative impliquant l'ensemble des membres de l'équipe, ainsi que des validations régulières auprès de nos enseignants spécialisés en analyse et en qualité de développement.

Le présent rapport aborde l'organisation interne de notre équipe ainsi que notre approche de modélisation des exigences du client, structurée comme suit :

\begin{enumerate}
    \item Diagrammes de cas d'utilisation
    \item Fiches descriptives
    \item Prototype d'interface
    \item Diagramme de classe
    \item Traçabilité globale
\end{enumerate}

Il est entendu que chaque section est itérative et a fait l'objet de plusieurs cycles de révision pour garantir la cohérence et la complétude de la modélisation.

\subsubsection{Analyse du diagramme de cas d'utilisation}

Le diagramme présenté ci-dessus modélise l'ensemble des interactions fonctionnelles du système Labyrinthe. Il identifie \textbf{sept cas d'utilisation principaux} répartis entre deux acteurs : l'Utilisateur et le Système.

\subsection{Acteurs identifiés}

\begin{itemize}
    \item \textbf{Utilisateur} : acteur principal, représente le joueur interagissant avec l'application (lancement du jeu, déplacement dans le labyrinthe, consultation de la progression)
    \item \textbf{Système} : acteur secondaire représentant les processus automatiques de persistance des données (sauvegarde de la progression sans intervention explicite de l'utilisateur)
\end{itemize}

La présence d'un acteur "Système" se justifie dans notre contexte par la nécessité de modéliser explicitement les opérations de persistance automatique. Cette modélisation permet de distinguer clairement les actions intentionnelles de l'utilisateur des mécanismes de sauvegarde implicites déclenchés par le système après chaque action significative (complétion d'un niveau, modification du profil).

\subsection{Cas d'utilisation}

Le diagramme couvre intégralement le \textbf{mode progression}, qui constitue le cœur fonctionnel de l'application :

\begin{enumerate}
    \item \textbf{Lancer en mode progression} : point d'entrée du mode progression, incluant la saisie du nom du joueur
    \item \textbf{Charger une partie} : reprise d'une progression existante pour un joueur connu
    \item \textbf{Jouer} : action générique d'interaction avec le jeu
    \item \textbf{Commencer une partie} : démarrage d'une nouvelle session de jeu
    \item \textbf{Se déplacer dans le labyrinthe} : déplacement case par case du joueur (cas d'utilisation détaillé dans la fiche "Déplacer joueur")
    \item \textbf{Afficher la progression} : consultation du tableau de bord avec niveaux complétés, score, caractéristiques
    \item \textbf{Sauvegarder la progression} : persistance automatique de l'état du joueur (acteur Système)
\end{enumerate}

\subsection{Clarifications}

à rephrase

\begin{itemize}
    \item Distinction nette entre "Charger une partie" pour un profil existant et "Lancer en mode progression" pour une nouvelle partie.
    \item Attribution explicite de la sauvegarde automatique au "Système".
    \item Différenciation entre l'action générale de "Jouer" et l'action spécifique de "Se déplacer dans le labyrinthe". Jouer englobe plus de fonctionnalités que le déplacement.
\end{itemize}

Les boutons "Retour" et "Quitter" ont une flèche d'arborescence prévisible qui n'est pas mentionnée afin de ne pas surcharger le prototype d'interface.

La fenêtre de progression est bien entendu liée au profil chargé ; que l'on reprenne une partie ou que l'on en crée une nouvelle, la page de progression sera adaptée aux données de progression du joueur (monde, défis complétés...).

Ce choix a été fait afin que le prototype comporte la visualisation de la progression ainsi que le choix du défi dans la même page ; deux pages différentes nécessiteraient plus d'arborescence (notamment un bouton visualiser progression...) résultant en moins de fluidité pour l'utilisateur.

Le parcours utilisateur se décompose en cinq étapes principales, chacune correspondant à un écran distinct :

Cette séquence garantit une progression logique et intuitive, sans étape superflue, conformément aux exigences du client.

\subsection{Structure générale}

L'architecture se décompose en trois niveaux conceptuels :

\begin{enumerate}

\item \textbf{Niveau contrôleur} : \texttt{Main} orchestre le flux général (lancement du jeu, sélection du mode, interaction avec la base de données)
\item \textbf{Niveau logique métier} : \texttt{GameMode}, \texttt{Maze}, \texttt{Player}, \texttt{PlayerProgress}, \texttt{Level}, \texttt{Challenge} encapsulent les règles métier et l'état de la partie
\item \textbf{Niveau persistance} : \texttt{PlayerDatabase} isole les opérations de sauvegarde et de chargement
\end{enumerate}

Cette séparation en couches garantit une maintenance facilitée et une testabilité accrue : chaque niveau peut être testé indépendamment, réduisant ainsi les risques d'anomalies et facilitant l'évolution future du système.

\subsection{Conformité avec la méthode vue en cours}

La méthodologie vue en \texttt{Analyse} préconise l'ordre suivant :

\begin{enumerate}
\item Une classe \texttt{Main} est créée
\item Elle comporte une fonction par cas d'utilisation
\item Pour chaque cas d'utilisation, on enrichit le diagramme de façon à la supporter
\end{enumerate}

Notre conception respecte cette méthodologie tout en l'adaptant aux spécificités du projet. 

\subsection{Correspondance avec les cas d'utilisation}

Afin de démontrer que l'ensemble des cas d'utilisation sont effectivement couverts par la classe \texttt{Main}, nous établissons la correspondance suivante :

\vspace{1em}
\begin{center}
\begin{tabular}{|p{5cm}|p{8cm}|}
\hline
\textbf{Fonctions du \texttt{Main}} & \textbf{Cas d'utilisation correspondants} \\
\hline
\texttt{startGame(): GameMode} & Lancer mode progression + Commencer une partie (englobe donc Jouer)\\
\hline
\texttt{movePlayerPosition(game: GameMode, direction: String)} & Se déplacer dans le labyrinthe \\
\hline

\hline

Cette approche évite la création de fonctions redondantes qui résulteraient d'une correspondance stricte entre chaque cas d'utilisation et chaque méthode. En effet, plusieurs cas d'utilisation partagent une logique commune et bénéficient d'un regroupement fonctionnel.

\subsection{Équilibre entre méthodologie et besoin client}

Notre modélisation représente un équilibre optimal entre le respect de la méthodologie d'\texttt{Analyse} et la satisfaction des exigences fonctionnelles exprimées dans le sujet.

En effet, une application strictement littérale de la méthodologie (une fonction par cas d'utilisation) aurait généré des fonctions superflues ou redondantes, notamment pour "Jouer" et "Commencer une partie", compte tenu de la structure actuelle de nos classes. 

À l'inverse, ignorer complètement la méthodologie aurait compromis la cohérence pédagogique du projet.

Notre solution concilie ces deux impératifs : les cas d'utilisation présentant un recouvrement fonctionnel ("Jouer", "Commencer une partie", "Lancer mode progression") sont unifiés au sein de la méthode \texttt{startGame(): GameMode}. 

L'utilisation d'un type abstrait \texttt{GameMode} et du polymorphisme permet de préserver la distinction nécessaire entre les différents modes de jeu tout en mutualisant le code commun.

\subsection{Clarifications}

La fonction \texttt{startGame()}, comme indiqué dans le tableau précédent, supporte plusieurs cas d'utilisation. Ce choix de conception mérite une explication détaillée :

Cette méthode retourne un \texttt{GameMode}, qui peut aussi bien être le mode libre que le mode progression (type abstrait hérité). Elle est conçue pour être appelée depuis le menu de départ. 

Ce type abstrait permet un traitement unifié des deux modes de jeu pour les opérations communes (par exemple, \texttt{movePlayerPosition}), évitant ainsi les duplications de code.

\begin{figure}[h!]
\centering
\includegraphics[width=0.8\textwidth]{figures/diagramme_de_classe/default/default/fig_2.png}
\caption{startGame() dans le Menu}
\label{fig:diagramme_de_classe_default_2}
\end{figure}

Elle sert donc à démarrer une partie, quel que soit le mode choisi, et couvre naturellement le cas d'utilisation "Lancer mode progression".

Dans le cas d'un lancement en mode progression, le joueur est dirigé vers une interface de saisie du nom. Une fois le nom validé, la fonction

\texttt{loadPlayer(name: String): Player} est invoquée afin de :
\begin{itemize}
\item Charger le profil existant si un joueur avec ce nom est enregistré dans la base de données, permettant ainsi de reprendre la progression sauvegardée
\item Créer un nouveau profil si aucun joueur avec ce nom n'existe, assurant une expérience fluide pour les nouveaux joueurs
\end{itemize}

Cette conception garantit une gestion cohérente et efficace du cycle de vie des profils joueurs.
